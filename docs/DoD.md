# Definition of Done (DoD)

A bucket of tasks or feature is only considered "Done" when it meets the following comprehensive criteria:

## 1. Code Quality & Standards
- Code compiles/runs without errors and warnings.
- Follows existing project conventions and linting rules.
- No dead code, temporary console prints, or debugging artifacts are left behind.
- Code is peer-reviewed and meets architectural cleanly standards.

## 2. Documentation & Code Comments
- Complex logic is properly commented and explained inline.
- Public functions, API routes, and components have updated docstrings.
- Project documentation (e.g., `README.md`, `docs/`) is updated to reflect structural or usage changes.

## 3. Quality Assurance (QA) & Quality Control (QC)
- Unit tests and regression tests are written or updated for all new functionality.
- All existing and new tests pass locally (e.g., via `unittest`).
- Manual verification is performed for specific edge cases and error states.

## 4. Versioning & Release Notes
- Appropriate version numbers are incremented as required.
- Release notes or changelogs are updated with clear, human-readable descriptions of features, bug fixes, and breaking changes.

## 5. Handover & Summary (Agent Work Rule)
- **When you are done with a bucket of tasks, always provide a full plain text summary that can be easily copied and pasted for the Architect to review.**
